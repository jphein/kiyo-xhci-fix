> >      stalled endpoint triggers an xHCI stop-endpoint timeout, and the
> >      kernel declares the host controller dead.
>
> Shouldn't a fix for this be added to xhci-hcd?  A funky device shouldn't
> be allowed to crash an entire USB bus.

You're right, and I agree that xhci-hcd is where the real fix belongs.

The problem is in xhci_handle_command_timeout() in xhci-ring.c.  When a
stop-endpoint command times out, the handler takes a different path than
other timed-out commands.  For most command types it attempts
xhci_abort_cmd_ring() first and only kills the HC if the abort itself
fails.  But for TRB_STOP_RING (stop-endpoint), it skips the abort
entirely and goes straight to xhci_hc_died():

	case TRB_STOP_RING:
		xhci_handle_command_timeout(ep, ...);
		xhci_warn(xhci,
			"xHCI host not responding to stop endpoint command\n");
		xhci_halt(xhci);
		xhci_hc_died(xhci);

There is also a TODO comment near the top of the command timeout handler
acknowledging the gap -- it notes the need to distinguish between
-ETIMEDOUT (command timed out, HC may still be alive) and -ENODEV (HC
genuinely gone), but that distinction is not yet implemented.

The xHCI spec provides a recovery ladder for exactly this situation:
Command Abort, then Port Reset or Disable Slot for the offending device,
then Reset Device, with controller-level recovery (LHCRST/HCRST) only as
a last resort.  The HCE bit in USBSTS can distinguish a true hardware
error from a command timeout.  FreeBSD's xHCI driver implements per-device
recovery along these lines; Linux currently does not.

Mathias Nyman looked at this in 2016 (in the context of a similar report
about stop-endpoint timeouts cascading into HC death) and noted the fix
was non-trivial due to deadlock risks in the command completion path.  It
has remained an open problem since.

The quirks in this series are intended as defense-in-depth -- they prevent
the device from reaching the state that triggers the stop-endpoint
timeout in the first place.  But they don't fix the underlying xhci-hcd
fragility, and I agree that a device-scoped recovery path in the
stop-endpoint timeout handler would be the proper long-term solution.

Would it make sense to submit the quirks now (to protect users hitting
this in the field) while the xhci-hcd improvement is worked on
separately?
